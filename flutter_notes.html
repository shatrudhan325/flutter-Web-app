1.1 Variables & Type System

Dart is a statically typed, type-inferred language. Every value in Dart is an object — even primitives like int and bool are instances of classes. This means you can call methods on any value, and everything inherits from the root Object class.

var lets Dart infer the type at compile time — once assigned, the type locks in. final is a runtime constant: set once, never reassigned (but the object itself can be mutable). const is a compile-time constant — the value must be known before the program runs and the object is deeply immutable. dynamic bypasses type checking entirely and is resolved at runtime — use sparingly.

Type inference keeps code concise while retaining all type-safety benefits. In real Flutter code, you'll use final most often for widget fields and local variables that don't change after initial assignment.

Dart 
// var — type inferred at compile time, then locked
var name = 'Flutter';          // inferred as String
var version = 3.19;             // inferred as double

// final — set once, known at runtime
final String platform = 'Mobile';
final startTime = DateTime.now();  // runtime value

// const — compile-time constant, deeply immutable
const double pi = 3.14159;
const maxRetries = 3;
const List<String> colors = ['red', 'green', 'blue'];

// dynamic — bypasses type checking (use sparingly)
dynamic anything = 42;
anything = 'now a string';   // allowed, no error
anything = true;              // allowed

// Type annotations (explicit)
int count = 0;
String message = 'Hello';
bool isLoggedIn = false;
List<int> scores = [95, 87, 72];
Map<String, dynamic> config = {'theme': 'dark', 'fontSize': 16};
💡 Best Practice

Prefer final by default. Use const for widget constructors and truly static values. Avoid dynamic unless parsing JSON or interfacing with untyped APIs — it defeats Dart's type safety.

1.2 Data Types & Collections

Dart provides three core collection types: List (ordered, duplicates allowed), Set (unordered, unique elements only), and Map (key-value pairs). All are generic — always specify the type parameter for type safety and better IDE support.

The spread operator (...) merges collections. Collection if and collection for let you build collections with conditional and iterative logic inline — extremely useful in Flutter widget trees. The cascade notation (..) lets you chain multiple operations on the same object without repeating the variable name.

Dart 
// List — ordered collection
List<String> fruits = ['apple', 'banana', 'mango'];
fruits.add('grape');
fruits.removeWhere((f) => f.startsWith('b'));

// Map — key-value pairs
Map<String, int> scores = {'Alice': 95, 'Bob': 87};
scores['Charlie'] = 91;
final top = scores.entries.reduce((a, b) => a.value > b.value ? a : b);

// Set — unique elements
Set<String> unique = {'a', 'b', 'a'}; // only 2 items

// Spread operator
List<String> moreFruits = ['pear', 'plum'];
List<String> allFruits = [...fruits, ...moreFruits];

// Collection if & for — used in Flutter widget trees!
bool isLoggedIn = true;
List<String> menu = [
  'Home',
  if (isLoggedIn) 'Profile',
  if (isLoggedIn) 'Logout' else 'Login',
  for (var item in ['Settings', 'Help']) item,
];

// Cascade notation (..) — chain operations
final buffer = StringBuffer()
  ..write('Hello')
  ..write(' ')
  ..write('Flutter');
print(buffer.toString()); // Hello Flutter
🔑 Collection if/for in Flutter

Collection if and for are essential in Flutter — they let you conditionally include widgets or build widget lists inline without needing external logic. This keeps your build() method clean and declarative.

1.3 Functions & Closures

Dart functions are first-class objects — they can be assigned to variables, passed as arguments, and returned from other functions. This enables powerful higher-order programming patterns used throughout Flutter (callbacks, builders, listeners).

Dart supports positional parameters (required by position), optional positional ([param]), named parameters ({param}), and combinations. The required keyword enforces named parameters. Closures capture their surrounding scope — a closure remembers variables from the function that created it, enabling patterns like factories and counters.

Avoid creating anonymous functions directly inside build() for performance-sensitive callbacks — they create new function objects each rebuild. Prefer referencing named methods or extracting the logic.

1.4 Null Safety

Dart's sound null safety (introduced in Dart 2.12) eliminates null reference exceptions at compile time. By default, no variable can hold null unless explicitly declared nullable with ?. The compiler statically analyzes all code paths to guarantee non-nullable variables always have a value.

The late keyword defers initialization — useful for values not available at construction time (e.g., dependency-injected services). The null assertion operator (!) overrides the type system — it tells Dart "I know this is non-null" and crashes if wrong. Use it only when you're certain.

Non-Nullable (default)

Must always have a value. Compiler guarantees it. Zero runtime null checks needed.

String name = 'Flutter';
// name = null; ← COMPILE ERROR
Nullable (with ?)

Can hold null. Requires null checks before use. Use null-aware operators.

String? nickname = null;
// safe to assign null
Dart 
String name = 'Flutter';      // non-nullable
String? nickname = null;       // nullable — OK

// Null-aware operators
String display = nickname ?? 'Anonymous';  // if null use default
int? length = nickname?.length;            // safe access — null if nickname is null
nickname ??= 'Default';                    // assign only if null

// Null-aware cascade
nickname?..trim()..toLowerCase(); // run cascade only if non-null

// late — must be assigned before first read
late String authToken;
void initAuth() {
  authToken = fetchToken(); // assigned before use
}

// late final — assigned exactly once
late final String userId;
userId = generateId(); // can only be set once

// Null assertion — use carefully!
String forced = nickname!; // throws if nickname is null

// Pattern: check before use
if (nickname != null) {
  print(nickname.length); // promoted to String (non-null)
}
🔑 Sound Null Safety

"Sound" means if the compiler says a variable is non-null, it's guaranteed at runtime — no exceptions. Mixed-mode (unsound) null safety exists in packages that haven't migrated, but your code remains safe.

1.5 OOP in Dart

Dart is a fully object-oriented language. Classes support multiple constructor types: default, named, factory, and const. Inheritance (extends) creates is-a relationships with single inheritance. Mixins (with) add capabilities without inheritance — perfect for cross-cutting concerns like serialization or logging. Interfaces (implements) enforce contracts without inheriting implementation.

Abstract classes define contracts with optional partial implementations. Extension methods add functionality to existing types without subclassing — one of Dart's most powerful features for writing expressive, clean code.

Dart 
import 'dart:math' show pi;

abstract class Shape {
  double get area;
  double get perimeter;
  String describe() => '${runtimeType}: area=${area.toStringAsFixed(2)}';
}

mixin Serializable {
  Map<String, dynamic> toJson();
}

class Circle extends Shape with Serializable {
  final double radius;
  const Circle({required this.radius});
  Circle.unit() : radius = 1.0;   // named constructor
  factory Circle.fromJson(Map<String, dynamic> j) =>
      Circle(radius: (j['radius'] as num).toDouble());

  @override double get area => pi * radius * radius;
  @override double get perimeter => 2 * pi * radius;
  @override Map<String, dynamic> toJson() =>
      {'type': 'circle', 'radius': radius};
}

// Extension methods — add to existing types
extension StringUtils on String {
  String get capitalized =>
      isEmpty ? '' : this[0].toUpperCase() + substring(1);
  bool get isEmail => contains('@') && contains('.');
  String get initials =>
      split(' ').map((w) => w.isEmpty ? '' : w[0]).join('').toUpperCase();
}

print('hello world'.capitalized); // Hello world
print('John Doe'.initials);       // JD
💡 Mixin vs Interface vs Extends

Use extends for a true is-a relationship. Use implements when you only want the contract (must implement all methods). Use with (mixins) for sharing behavior across unrelated classes — like a Serializable capability.

1.6 Async Programming

Dart runs on a single-threaded event loop, but async programming allows non-blocking I/O. The event loop processes two queues: the microtask queue (higher priority — Future.value, scheduleMicrotask) and the event queue (I/O, timers, user events). Microtasks always drain before the next event.

Future represents a single async value — uncompleted, completed with data, or completed with error. async/await is syntactic sugar over Future chains. Stream represents a sequence of async events over time. Single-subscription streams (like HTTP responses) differ from broadcast streams (like user events) which allow multiple listeners.

Dart 
// Future — single async value
Future<String> fetchData() async {
  await Future.delayed(Duration(seconds: 2));
  return 'Hello from server!';
}

// Error handling in async
Future<User> getUser(int id) async {
  try {
    final response = await http.get(
      Uri.parse('https://api.example.com/users/$id'),
    );
    if (response.statusCode == 200) {
      return User.fromJson(jsonDecode(response.body));
    }
    throw HttpException('Status: ${response.statusCode}');
  } on SocketException {
    throw NetworkException('No internet connection');
  } catch (e) {
    throw UnexpectedException(e.toString());
  }
}

// Stream — sequence of async values
Stream<int> countdown(int from) async* {
  for (int i = from; i >= 0; i--) {
    await Future.delayed(Duration(seconds: 1));
    yield i; // emit value to stream
  }
}

// Listen to stream
countdown(5).listen(
  (value) => print('Count: $value'),
  onDone: () => print('Blast off!'),
  onError: (e) => print('Error: $e'),
);
⚠️ Async Gap & Context Safety

After an await, always check if (mounted) in StatefulWidget before using context. The widget may have been removed from the tree during the async operation, making the context invalid.

1.7 Memory Management & Garbage Collection

Dart uses an advanced, generational Garbage Collector (GC) designed specifically for Flutter's fast UI rendering architecture.

New Space (Nursery)

Most objects in Flutter (like local variables and pure UI Widgets that rebuild every frame) die young. The Nursery is small and cleaned out constantly in "scavenge" phases. This phase is extremely fast and specifically optimized to run in the micro-seconds of idle time between flutter frames, preventing any stuttering.

Old Space

Objects that survive a scavenge (like long-lived application state, singletons, or heavy cached images) are moved to the Old Space. Cleaning this uses a "Mark-Sweep" algorithm and is executed much less frequently because it is a heavier operation.

Pro TipWhy it matters for Memory
Use const constructorsconst Widgets are allocated exactly once in a special immutable memory area. They never hit the Nursery and the GC never has to clean them up.
Call dispose()Controllers (TextEditingController, AnimationController) wrap native C++ handlers. If you forget to dispose them, Dart's GC won't free the native memory, causing a leak.
Avoid Global VariablesGlobals stay in Old Space forever until the App dies. Prefer Scoped State.

2.1 How Flutter Works

Flutter is a multi-platform UI toolkit that does not rely on native UI components. Instead it uses its own high-performance rendering engine (Skia or the newer Impeller) to paint every pixel directly onto a canvas. This means Flutter widgets look and behave identically on every platform — Android, iOS, Web, Windows, macOS, and Linux.

The framework is organized into four conceptual layers. Your Dart code sits at the top. The Flutter Framework provides Material, Cupertino, and the base widget library. The Flutter Engine handles rendering, text layout, and the Dart runtime. The Platform Embedder integrates with each native OS to provide the canvas, input events, and plugin channels.

In debug mode, Dart uses JIT (Just-In-Time) compilation for hot reload. In release mode, Dart AOT (Ahead-Of-Time) compiles to native machine code — producing fast, small binaries with no interpreter overhead.

💡 Impeller — The Future Renderer

Impeller replaces Skia as Flutter's default renderer on iOS (stable) and Android (beta). It pre-compiles shaders at build time to eliminate shader compilation jank — a common complaint with Skia's on-demand shader compilation.

2.2 Widget Tree & Three Trees

Flutter has three distinct trees that work in concert. The Widget tree is immutable — widgets are lightweight configuration objects. When setState() is called, Flutter rebuilds the widget tree from that point down. The Element tree is the live instance tree — Elements persist across rebuilds and reconcile the old widget tree with the new one. RenderObjects do the actual layout and painting.

This architecture means: rebuilding widgets is cheap (they're just Dart objects), but relayout and repaint are expensive. Flutter only repaints when the render object truly needs to change. BuildContext is actually the Element — it represents the widget's position in the tree.

🔑 Widget = Config, Element = Instance

Never confuse widgets and elements. Widget describes what to build — it's disposable. Element manages lifecycle and holds state. When you call setState(), Flutter only reconstructs the Widget tree; the Element tree is reconciled, and only changed RenderObjects repaint.

2.3 Stateless vs Stateful Widgets

A StatelessWidget is a pure function of its configuration — given the same inputs, it always produces the same UI. It has no internal mutable state. This makes it predictable, easy to test, and highly performant. Use it for static content, display-only components, and layouts that don't change.

A StatefulWidget creates a separate State object that persists across rebuilds. The State holds mutable data and can trigger rebuilds via setState(). The lifecycle is critical to understand: initState() for one-time setup, build() for rendering, didUpdateWidget() when parent changes configs, and dispose() for cleanup.

Flutter 
// StatelessWidget — pure function of config
class WelcomeCard extends StatelessWidget {
  final String username;
  const WelcomeCard({super.key, required this.username});

  @override
  Widget build(BuildContext context) {
    return Card(
      child: Padding(
        padding: const EdgeInsets.all(16),
        child: Text('Welcome, $username!',
          style: Theme.of(context).textTheme.headlineSmall),
      ),
    );
  }
}

// StatefulWidget — mutable state
class ToggleButton extends StatefulWidget {
  const ToggleButton({super.key});
  @override
  State<ToggleButton> createState() => _ToggleButtonState();
}

class _ToggleButtonState extends State<ToggleButton> {
  bool _isOn = false;

  @override
  void initState() {
    super.initState(); // always call super first
    // one-time setup: controllers, listeners, etc.
  }

  @override
  Widget build(BuildContext context) {
    return ElevatedButton(
      onPressed: () => setState(() => _isOn = !_isOn),
      style: ElevatedButton.styleFrom(
        backgroundColor: _isOn ? Colors.green : Colors.grey,
      ),
      child: Text(_isOn ? 'ON' : 'OFF'),
    );
  }

  @override
  void dispose() {
    // dispose controllers, cancel subscriptions
    super.dispose(); // always call super last
  }
}

2.4 BuildContext In Depth

BuildContext is literally the Element — your widget's position in the tree. It enables widgets to communicate upward through the tree to find ancestors. Theme.of(context), MediaQuery.of(context), Navigator.of(context) all traverse up the Element tree to find the nearest matching ancestor.

Contexts are only valid within their build scope. After an await, the widget may have been disposed — always check mounted before using context post-async. Never store context in a field of a State object for async use without checking mounted.

Flutter 
// Safe context usage after async
class _LoginPageState extends State<LoginPage> {
  bool _loading = false;

  Future<void> _handleLogin() async {
    setState(() => _loading = true);
    try {
      await authService.login(email, password);
      // ✅ Check mounted after await!
      if (!mounted) return;
      Navigator.of(context).pushReplacement(
        MaterialPageRoute(builder: (_) => const HomeScreen()),
      );
    } catch (e) {
      if (!mounted) return;
      ScaffoldMessenger.of(context).showSnackBar(
        SnackBar(content: Text(e.toString())),
      );
    } finally {
      if (mounted) setState(() => _loading = false);
    }
  }

  @override
  Widget build(BuildContext context) {
    // Access theme, size, navigator from context
    final theme = Theme.of(context);
    final size = MediaQuery.sizeOf(context);
    print('Screen: ${size.width}x${size.height}');
    return Scaffold(
      backgroundColor: theme.colorScheme.surface,
      body: Center(child: Text('Login')),
    );
  }
}
⚠️ Context Scope

Avoid calling Navigator.of(context) or ScaffoldMessenger.of(context) inside initState() — the widget isn't in the tree yet. Use WidgetsBinding.instance.addPostFrameCallback() or move to didChangeDependencies().

2.5 Element vs RenderObject (The Three Trees)

We touched on this in 2.2, but tracking the lifecycle reveals why Flutter is so fast:

1. Widget Tree (Config)

You write this. It represents the immutable configurations. These are destroyed and recreated constantly (e.g. 60 times a second during an animation). It costs essentially nothing to create Dart classes.

2. Element Tree (Lifecycle)

Flutter builds this automatically. This is the mutable instantiation of a widget at a specific location in the UI. If you rebuild a Widget, Flutter compares it to the corresponding Element. If the type and Key are the same, the Element is kept and just updated.

3. RenderObject Tree (Pixels)

The Element instantiates a RenderObject. This handles measuring constraints (layout) and drawing pixels (paint). Relayouting and repainting are the expensive operations that cause jank.

3.1 Container & Box Model

Container is Flutter's swiss-army widget — it combines padding, margin, decoration, constraints, alignment, and transforms in one. When given no child, Container fills available space. With a child, it sizes to the child unless constrained. Never use both color and decoration simultaneously — decoration subsumes color.

BoxDecoration unlocks rich styling: gradients, border radius, box shadows, and images. The constraints property defines min/max width and height — useful for making widgets flexible yet bounded. Transform applies matrix transformations (rotate, scale, translate) without affecting layout.

Flutter 
Container(
  margin: const EdgeInsets.all(16),
  padding: const EdgeInsets.symmetric(horizontal: 24, vertical: 16),
  width: double.infinity,
  constraints: const BoxConstraints(
    minHeight: 100,
    maxHeight: 300,
  ),
  decoration: BoxDecoration(  // ← use decoration, NOT color
    gradient: const LinearGradient(
      colors: [Color(0xFF667eea), Color(0xFF764ba2)],
      begin: Alignment.topLeft,
      end: Alignment.bottomRight,
    ),
    borderRadius: BorderRadius.circular(16),
    boxShadow: [
      BoxShadow(
        color: Colors.black.withOpacity(0.3),
        blurRadius: 12,
        offset: const Offset(0, 6),
      ),
    ],
    border: Border.all(color: Colors.white24, width: 1),
  ),
  child: const Text(
    'Styled Container',
    style: TextStyle(color: Colors.white, fontSize: 18),
  ),
)
⚠️ color vs decoration

Setting both color and decoration on a Container throws an assertion error at runtime. When using BoxDecoration, place your color inside BoxDecoration(color: ...) instead.

3.2 Row & Column Deep Dive

Row lays children horizontally; Column lays them vertically. MainAxisAlignment controls spacing along the main axis. CrossAxisAlignment aligns children on the perpendicular axis. Expanded fills all remaining main-axis space. Flexible can fill some space proportionally via flex factor. Spacer is a convenient shorthand for a flexible empty space.

Row overflow (children wider than available width) crashes with an overflow error. Fix it by wrapping children in Expanded, using Flexible, or wrapping the Row in a SingleChildScrollView with horizontal scroll.

Flutter 
// Row with alignment and Expanded
Row(
  mainAxisAlignment: MainAxisAlignment.spaceBetween,
  crossAxisAlignment: CrossAxisAlignment.center,
  children: [
    const Icon(Icons.flutter_dash, size: 32),
    const Expanded(  // fills remaining space
      child: Text('Flutter is awesome', overflow: TextOverflow.ellipsis),
    ),
    const Spacer(),  // flexible gap
    ElevatedButton(onPressed: () {}, child: const Text('Go')),
  ],
)

// Column with flex proportions
Column(
  children: [
    Flexible(flex: 2, child: Container(color: Colors.blue)),   // 2/5 height
    Flexible(flex: 3, child: Container(color: Colors.green)),  // 3/5 height
  ],
)

// Fix Row overflow with Flexible
Row(
  children: [
    const Icon(Icons.star),
    const Flexible(  // shrinks to fit, won't overflow
      child: Text('Very long text that might overflow without Flexible',
        overflow: TextOverflow.ellipsis),
    ),
  ],
)
💡 Expanded vs Flexible

Expanded is Flexible with fit: FlexFit.tight — it must fill the space. Flexible defaults to FlexFit.loose — it can use up to the flex space but can be smaller. Use Flexible when the child might be smaller than its allocation.

3.3 Stack & Positioned

Stack layers children on the Z-axis — later children are painted on top. Non-Positioned children use the alignment property of the Stack. Positioned lets you place children at exact offsets from the Stack's edges. Positioned.fill is a shorthand for covering the entire Stack area.

Common use cases: badge overlays on icons, gradient overlays on images, floating labels, bottom-of-card CTAs, and custom FAB-like components not tied to Scaffold.

Flutter 
Stack(
  children: [
    // Base: full-size image
    Image.network(
      'https://picsum.photos/400/200',
      fit: BoxFit.cover,
      width: double.infinity,
      height: 200,
    ),
    // Gradient overlay — fades bottom to black
    Positioned.fill(
      child: const DecoratedBox(
        decoration: BoxDecoration(
          gradient: LinearGradient(
            begin: Alignment.topCenter,
            end: Alignment.bottomCenter,
            colors: [Colors.transparent, Colors.black87],
          ),
        ),
      ),
    ),
    // Caption at bottom
    const Positioned(
      bottom: 16, left: 16, right: 16,
      child: Text('Mountain Landscape',
        style: TextStyle(color: Colors.white, fontSize: 18, fontWeight: FontWeight.bold)),
    ),
    // Badge top-right
    const Positioned(
      top: 8, right: 8,
      child: Chip(
        label: Text('NEW', style: TextStyle(fontSize: 11)),
        backgroundColor: Colors.purple,
      ),
    ),
  ],
)
🔑 Stack Sizing

By default, Stack sizes to its largest non-Positioned child. Set fit: StackFit.expand to force the Stack to fill its parent. All Positioned children are then placed relative to that expanded area.

3.4 ListView & GridView

ListView is a scrollable column. ListView(children: [...]) builds all children upfront — fine for short lists. ListView.builder is lazy — it only builds widgets currently visible on screen. For lists with thousands of items, builder is essential. ListView.separated adds dividers automatically between items.

GridView.count creates a fixed-column grid. GridView.builder is the lazy version for large grids. Never put a ListView inside a Column without either Expanded, shrinkWrap: true (avoid in long lists — expensive!), or a fixed height constraint.

Flutter 
// ListView.builder — lazy, efficient
ListView.builder(
  itemCount: messages.length,
  itemBuilder: (context, index) {
    final msg = messages[index];
    return MessageTile(message: msg, key: ValueKey(msg.id));
  },
)

// ListView.separated — with dividers
ListView.separated(
  itemCount: items.length,
  separatorBuilder: (_, __) => const Divider(height: 1),
  itemBuilder: (context, index) => ListTile(
    title: Text(items[index].title),
    subtitle: Text(items[index].subtitle),
    leading: CircleAvatar(child: Text('${index + 1}')),
    onTap: () => onItemTap(items[index]),
  ),
)

// GridView.builder — lazy 2-column product grid
GridView.builder(
  gridDelegate: const SliverGridDelegateWithFixedCrossAxisCount(
    crossAxisCount: 2,
    crossAxisSpacing: 12,
    mainAxisSpacing: 12,
    childAspectRatio: 0.75,
  ),
  itemCount: products.length,
  itemBuilder: (context, index) =>
      ProductCard(product: products[index]),
)
⚠️ shrinkWrap: true

shrinkWrap: true forces ListView to measure all children even if they're not visible — defeating the lazy loading entirely. Instead, wrap ListView in Expanded or give it a fixed height with a SizedBox.

3.5 Responsive Layouts

MediaQuery provides screen dimensions, padding, orientation, and device pixel ratio. Use MediaQuery.sizeOf(context) (Flutter 3.10+) instead of MediaQuery.of(context).size — the former only rebuilds when size changes, reducing unnecessary rebuilds.

LayoutBuilder gives you the parent widget's constraints — better than MediaQuery when you need to respond to container size rather than screen size. This is essential for reusable components that might appear in sidebars or dialogs. SafeArea ensures content isn't hidden behind notches, status bars, or system gesture areas.

5.1 Navigator 1.0 — Imperative

Navigator maintains a stack of Route objects. push() adds a route on top (new screen forward). pop() removes the top route (goes back). pushReplacement replaces the current route — use it after login to prevent the user going "back" to the login screen. pushAndRemoveUntil clears the stack — useful for logout flows.

Routes can return data — push() returns a Future that resolves when the pushed route is popped. Pass data via constructor arguments. WillPopScope intercepts the back button to show confirmation dialogs or prevent accidental exits.

Flutter

  MaterialPageRoute(builder: (_) => const ColorPickerScreen()),
);
if (selectedColor != null && mounted) {
  _applyColor(selectedColor);
}

// Pop with result
Navigator.of(context).pop('red'); // returns 'red' to awaiting push

// After login — clear all back stack
Navigator.of(context).pushAndRemoveUntil(
  MaterialPageRoute(builder: (_) => const HomeScreen()),
  (route) => false, // remove all routes below
);

// Custom page transition

6.1 setState — Local State

setState is Flutter's built-in mechanism for triggering a widget rebuild. It's perfect for local UI state — toggles, form input, counters, tab selections — that doesn't need to be shared with other widgets. The rule: if only one widget needs the state, keep it local.

Call setState() with the minimal change needed. Never put expensive operations inside setState()'s callback — only state mutations go there. To optimize, extract child widgets as const or separate StatelessWidgets so they don't rebuild when the parent calls setState().

6.2 Provider

Provider is built on InheritedWidget. A ChangeNotifier class holds state and calls notifyListeners() to trigger rebuilds. context.watch<T>() subscribes to changes and rebuilds on every notification. context.read<T>() reads the value once without subscribing — use it in callbacks and event handlers. context.select<T,S>() subscribes only to a specific piece of the model.

Flutter 
// ChangeNotifier model
class CartProvider extends ChangeNotifier {
  final List<CartItem> _items = [];

  List<CartItem> get items => List.unmodifiable(_items);
  int get itemCount => _items.length;
  double get total => _items.fold(0, (sum, i) => sum + i.price * i.qty);

  void addItem(Product product) {
    final idx = _items.indexWhere((i) => i.id == product.id);
    if (idx >= 0) { _items[idx].qty++; }
    else { _items.add(CartItem.fromProduct(product)); }
    notifyListeners();  // triggers all watchers
  }

  void removeItem(String id) {
    _items.removeWhere((i) => i.id == id);
    notifyListeners();
  }
}

// Provide at root
ChangeNotifierProvider(create: (_) => CartProvider(), child: const MyApp())

// In widgets:
final total = context.watch<CartProvider>().total;   // rebuilds on change
context.read<CartProvider>().addItem(product);        // no rebuild

// Selector — only rebuilds when itemCount changes
Selector<CartProvider, int>(
  selector: (_, cart) => cart.itemCount,
  builder: (_, count, __) => CartBadge(count: count),
)
⚠️ context.watch in Callbacks

Never use context.watch inside button onPressed or other callbacks — calling it outside build() will throw. Use context.read for event handlers. The rule: watch = inside build, read = inside callbacks.

6.3 Riverpod — Modern Approach

Riverpod fixes Provider's key limitations: no BuildContext required, providers are globally accessible and testable, no ProviderNotFoundException, and autoDispose cleans up providers when no longer needed. Providers are defined at top-level — they are not part of the widget tree.

Provider types form a hierarchy: Provider (read-only computed value) → StateProvider (simple mutable state) → NotifierProvider (complex state with methods) → FutureProvider (async data) → StreamProvider (live data stream). .family makes a provider take a parameter. .autoDispose disposes when last listener unsubscribes.

Flutter 
// Simple state — theme toggle
final themeProvider = StateProvider<ThemeMode>(
  (ref) => ThemeMode.system);

// Complex state with logic
class AuthNotifier extends Notifier<AuthState> {
  @override
  AuthState build() => const AuthState.initial();

  Future<void> login(String email, String pass) async {
    state = const AuthState.loading();
    try {
      final user = await ref.read(authRepoProvider).login(email, pass);
      state = AuthState.authenticated(user);
    } catch (e) {
      state = AuthState.error(e.toString());
    }
  }
  void logout() {
    ref.read(authRepoProvider).logout();
    state = const AuthState.initial();
  }
}
final authProvider = NotifierProvider<AuthNotifier, AuthState>(AuthNotifier.new);

// FutureProvider with autoDispose + family
final userProvider = FutureProvider.autoDispose
    .family<User, int>((ref, userId) async {
  return ref.read(userRepoProvider).getUser(userId);
});

// In ConsumerWidget — no BuildContext needed for providers
class ProfilePage extends ConsumerWidget {
  final int userId;
  const ProfilePage({super.key, required this.userId});
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final userAsync = ref.watch(userProvider(userId));
    return userAsync.when(
      data: (user) => UserCard(user: user),
      loading: () => const CircularProgressIndicator(),
      error: (e, _) => ErrorWidget(e),
    );
  }
}
🔑 ref.watch vs ref.read vs ref.listen

ref.watch subscribes and rebuilds on change (use in build). ref.read reads once without subscribing (use in callbacks). ref.listen triggers a side effect callback on change without rebuilding (use for navigation, snackbars).

6.4 BLoC Pattern

The Business Logic Component pattern strictly separates UI from business logic. Events are inputs (user actions, lifecycle events). States are outputs (what the UI renders). The Bloc processes events and emits states asynchronously. This makes business logic fully testable without Flutter — just Dart classes.

Cubit is a simplified BLoC without events — it exposes methods that emit states directly. Use Cubit when you don't need event history or complex transformations. Use BLoC when you need event debouncing, event transformation, or event logging middleware (EventTransformer).

Flutter 
// Events
abstract class AuthEvent {}
class LoginRequested extends AuthEvent {
  final String email, password;
  LoginRequested({required this.email, required this.password});
}
class LogoutRequested extends AuthEvent {}

// States (sealed class in Dart 3+)
sealed class AuthState {}
class AuthInitial extends AuthState {}
class AuthLoading extends AuthState {}
class AuthAuthenticated extends AuthState { final User user; AuthAuthenticated(this.user); }
class AuthError extends AuthState { final String message; AuthError(this.message); }

// BLoC
class AuthBloc extends Bloc<AuthEvent, AuthState> {
  final AuthRepository _repo;
  AuthBloc(this._repo) : super(AuthInitial()) {
    on<LoginRequested>(_onLogin);
    on<LogoutRequested>(_onLogout);
  }

  Future<void> _onLogin(LoginRequested event, Emitter<AuthState> emit) async {
    emit(AuthLoading());
    try {
      final user = await _repo.login(event.email, event.password);
      emit(AuthAuthenticated(user));
    } catch (e) { emit(AuthError(e.toString())); }
  }
  void _onLogout(_, __) { _repo.logout(); emit(AuthInitial()); }
}

// In widget — BlocConsumer combines Builder + Listener
BlocConsumer<AuthBloc, AuthState>(
  listener: (ctx, state) {
    if (state is AuthAuthenticated) ctx.go('/home');
    if (state is AuthError) ScaffoldMessenger.of(ctx).showSnackBar(SnackBar(content: Text(state.message)));
  },
  builder: (ctx, state) => switch (state) {
    AuthLoading() => const CircularProgressIndicator(),
    _ => LoginForm(),
  },
)
💡 Cubit vs BLoC

Use Cubit for simple state machines (loading/loaded/error). Use Bloc when you need event debouncing (search-as-you-type), event transformation, or need to log/replay events for analytics.

6.4 Architectural Comparison: Riverpod vs BLoC

Choosing between the two most popular state management solutions dictates your app's architecture.

Riverpod (Functional)

Created by the author of Provider. It solves Provider's biggest flaw: ProviderNotFoundException at runtime. Riverpod is completely compile-time safe. It relies on global constant declarations that act as "keys" to state. It heavily promotes functional programming paradigms and immutable state.

  • ✅ No BuildContext needed to read state
  • ✅ Safe merging/combining of states
  • ❌ Can be chaotic if globalProviders aren't structured well
BLoC (Event-Driven)

Created by Felix Angelov (formerly at Google). It enforces a strict unidirectional data flow: UI dispatches Events ➡️ BLoC processes Events ➡️ BLoC emits State. It heavily relies on the Widget tree (InheritedWidget) for scoping.

  • ✅ Extremely predictable and traceable

7.1 HTTP & Dio

The built-in http package handles simple GET/POST requests. Dio is the production-ready HTTP client — it adds interceptors (middleware for every request/response), automatic retries, FormData uploads, download progress, and request cancellation via CancelToken.

The interceptor pattern is critical for auth: attach the Bearer token on every request, and if the server returns 401, automatically refresh the token and retry the original request — all transparent to the rest of the app. BaseOptions sets global timeouts and headers.

Dart 
class DioClient {
  late final Dio _dio;

  DioClient() {
    _dio = Dio(BaseOptions(
      baseUrl: 'https://api.example.com/v1',
      connectTimeout: const Duration(seconds: 10),
      receiveTimeout: const Duration(seconds: 30),
      headers: {'Content-Type': 'application/json'},
    ));
    _dio.interceptors.addAll([
      _AuthInterceptor(_dio),
      LogInterceptor(requestBody: true, responseBody: true),
    ]);
  }

  Future<T> get<T>(String path, {
    Map<String, dynamic>? params,
    required T Function(dynamic) parser,
  }) async {
    final res = await _dio.get(path, queryParameters: params);
    return parser(res.data);
  }
}

// Auth interceptor — auto-attach + refresh token
class _AuthInterceptor extends Interceptor {
  final Dio dio;
  _AuthInterceptor(this.dio);

  @override
  void onRequest(RequestOptions options, RequestInterceptorHandler handler) {
    final token = TokenStorage.getAccessToken();
    if (token != null) options.headers['Authorization'] = 'Bearer $token';
    handler.next(options);
  }

  @override
  void onError(DioException err, ErrorInterceptorHandler handler) async {
    if (err.response?.statusCode == 401) {
      await TokenStorage.refreshToken();
      return handler.resolve(await dio.fetch(err.requestOptions));
    }
    handler.next(err);
  }
}
💡 dio vs http

Use http for simple one-off requests in scripts or small apps. Use Dio when you need interceptors (auth), retries, FormData, multipart uploads, download progress tracking, or request cancellation.

7.4 Repository Pattern

The Repository pattern abstracts data sources behind a clean interface. The UI layer only knows about the repository interface — it doesn't care whether data comes from an API, cache, or local database. This enables easy mocking in tests and swapping data sources without touching UI code.

Dart 
// Abstract interface (domain layer)
abstract interface class UserRepository {
  Future<User> getUser(int id);
  Future<List<User>> getUsers({int page = 1});
  Future<void> saveUser(User user);
}

// Implementation (data layer) with cache-then-network
class UserRepositoryImpl implements UserRepository {
  final UserRemoteDataSource _remote;
  final UserLocalDataSource _local;

  UserRepositoryImpl(this._remote, this._local);

  @override
  Future<User> getUser(int id) async {
    final cached = await _local.getUser(id);
    if (cached != null) return cached;  // serve from cache
    final user = await _remote.fetchUser(id);
    await _local.saveUser(user);         // cache result
    return user;
  }
}
💡 Test the Repository Contract

With an abstract interface, you can create a MockUserRepository in tests that returns hardcoded data. Your BLoC/Notifier tests never need a real network connection — they just test logic against the mock.

7.4 Advanced Networking: Dio Interceptors

For production apps, http package is often too basic. The dio package is the industry standard because of Interceptors. Interceptors act as middlewares for all outgoing requests and incoming responses. You can use them to automatically attach Auth Tokens, log requests, or seamlessly refresh expired tokens.

Dart 
final dio = Dio(BaseOptions(baseUrl: 'https://api.example.com/'));

dio.interceptors.add(InterceptorsWrapper(
  // 1. ON REQUEST (Inject Token)
  onRequest: (options, handler) async {
    final token = await secureStorage.read(key: 'token');
    if (token != null) {
      options.headers['Authorization'] = 'Bearer $token';
    }
    return handler.next(options); // Continue the request
  },
  
  // 2. ON RESPONSE (Log or Parse)
  onResponse: (response, handler) {
    print('Data Received: ${response.statusCode}');
    return handler.next(response); 
  },
  
  // 3. ON ERROR (Handle Token Expiration & Refresh)
  onError: (DioException e, handler) async {
    if (e.response?.statusCode == 401) {
      // Token expired! Attempt to refresh it
      final newToken = await refreshAuthToken();
      if (newToken != null) {
        // Retry the original failed request
        e.requestOptions.headers['Authorization'] = 'Bearer $newToken';
        final retryResponse = await dio.fetch(e.requestOptions);
        return handler.resolve(retryResponse);
      }
    }
    return handler.next(e); // Pass error down if not handled
  },
));

8.1 SharedPreferences

SharedPreferences provides a persistent key-value store backed by Android SharedPreferences and iOS NSUserDefaults. It survives app restarts and supports bool, int, double, String, and List<String>. Access is async — always await the instance. Use it for app settings, onboarding flags, and theme preferences. Never store sensitive data (tokens, passwords) here — it's stored as plain text.

8.4 Secure Storage

flutter_secure_storage stores data in the iOS Keychain and Android EncryptedSharedPreferences (backed by Android Keystore). This is cryptographically protected and not accessible without the app's credentials. Always store auth tokens, refresh tokens, and passwords here — never in SharedPreferences or Hive.

Dart 
class SecureStorageService {
  static const _storage = FlutterSecureStorage(
    aOptions: AndroidOptions(encryptedSharedPreferences: true),
    iOptions: IOSOptions(accessibility: KeychainAccessibility.first_unlock),
  );

  static const _accessKey  = 'access_token';
  static const _refreshKey = 'refresh_token';

  static Future<void> saveTokens(String access, String refresh) async {
    await Future.wait([
      _storage.write(key: _accessKey, value: access),
      _storage.write(key: _refreshKey, value: refresh),
    ]);
  }

  static Future<String?> getAccessToken() =>
      _storage.read(key: _accessKey);
  static Future<String?> getRefreshToken() =>
      _storage.read(key: _refreshKey);

  static Future<void> clearTokens() async {
    await Future.wait([
      _storage.delete(key: _accessKey),
      _storage.delete(key: _refreshKey),
    ]);
  }
}
🔑 Security Rule #1

The golden rule: if it's secret, use FlutterSecureStorage. Auth tokens, API keys, PINs, biometric data — all go in the secure enclave, not plain storage. On Android, set encryptedSharedPreferences: true to use EncryptedSharedPreferences backed by the Android Keystore.

9.1 Isolates & True Multithreading

Dart code runs in an Isolate, which has its own memory heap and a single thread running an Event Loop. Heavy synchronous code (like parsing a 10MB JSON file or applying image filters) blocks the Event Loop, causing UI "jank" (dropped frames). To fix this, you spawn a new Isolate. Isolates do not share memory; they communicate by passing messages via SendPort and ReceivePort.

Single Threaded (Blocking)

Main Isolate handles UI, touches, and async I/O. A heavy computation here will freeze the app until it finishes.

// UI freezes until done!
List<User> users = parseLargeJson(data);
Multi Isolate (Non-blocking)

Heavy work is offloaded to a background Isolate. The main Isolate is free to render 60/120fps animations.

// UI remains perfectly smooth
List<User> users = await Isolate.run(
  () => parseLargeJson(data)
);

Future<List<Product>> fetchAndParseProducts() async {

// Use when you need a long-running background worker
void startWorker() async {
  final receivePort = ReceivePort();
  await Isolate.spawn(_heavyWorker, receivePort.sendPort);

  // Listen to messages from the worker
  receivePort.listen((message) {
    if (message is SendPort) {
      message.send('Do task A'); // Send instruction to worker
    } else {
      print('Worker result: $message');
    }
  });
}

static void _heavyWorker(SendPort mainSendPort) {
  final workerReceivePort = ReceivePort();
  mainSendPort.send(workerReceivePort.sendPort); // Handshake

  workerReceivePort.listen((message) {
    // Process heavy task based on message
    final result = calculateFibonacci(50); 
    mainSendPort.send(result);
  });
}

10.1 Clean Architecture Deep Dive

A "Big Ball of Mud" where API calls, business logic, and UI are mixed together in StatefulWidgets makes the app impossible to maintain or test. Clean Architecture divides responsibilities into independent layers. The central principle is the Dependency Rule: source code dependencies must only point inwards.

  • Domain Layer: The heart of the software. Contains pure Dart Entities (like User) and UseCases (like LoginUserUseCase). It knows absolutely nothing about HTTP, JSON, or Flutter UI. It defines abstract UserRepository interfaces.
  • Data Layer: Contains the implementation of the Domain's Repositories. It fetches data via DataSources (API or local DB), parses JSON into Models, and converts them to pure Entities.
  • Presentation Layer: Knows only about the Domain layer. State managers (Bloc/Riverpod) call UseCases, get Entities back, and update local state to trigger Widget rebuilds.
Dart (Dependency Inversion) 
// 1. DOMAIN LAYER
class User { final String name; const User(this.name); }

abstract class AuthRepository {
  Future<User> login(String email, String pass);
}

class LoginUseCase {
  final AuthRepository repo;
  const LoginUseCase(this.repo); // Expects interface, not implementation!

  Future<User> execute(String e, String p) {
    if (!e.contains('@')) throw Exception('Invalid email');
    return repo.login(e, p);
  }
}

// 2. DATA LAYER
class AuthRepositoryImpl implements AuthRepository {
  final ApiDataSource api;
  AuthRepositoryImpl(this.api);

  @override
  Future<User> login(String email, String pass) async {
    final jsonStr = await api.post('/login', {'e':email, 'p':pass});
    // Parse JSON model, convert to pure User entity, return it
    return UserModel.fromJson(jsonStr).toEntity();
  }
}
